## 联调路径约定（E-6.1，非缺陷登记）

- **浏览器（Vite 开发）**：前端 **`http-client`** 使用 **`VITE_API_BASE_URL`**（常见为 **`/api`**），请求形如 **`/api/page-templates`**；由 **Vite 代理** 转发到后端 **根路径上的业务路由**（实际以 `vite.config` 为准）。

## 1. menu — 移动/排序

- **模块**：menu
- **接口**：`POST /menus/{id}/move`
- **请求方式**：POST
- **请求体**：按文档（实际以联调抓包为准）
- **期望**：`success: true`，菜单顺序持久化成功
- **实际**：HTTP 200 + `success: false` + `code: SYSTEM_ERROR` + `message: 系统繁忙，请稍后再试`
- **影响**：菜单移动/排序无法完成真实成功路径
- **建议后端修复方式**：排查服务端异常栈与事务/约束，修复后返回标准成功信封
- **前端当前处理方式**：真实失败 throw，`ElMessage` 提示错误，不假装成功

## 4. page-version — 详情查询

- **模块**：page-version
- **接口**：`GET /page-versions/{id}`
- **请求方式**：GET
- **请求体**：无
- **期望**：`success: true`，返回 `page_version` 详情
- **实际**（本机 `127.0.0.1:9876`，示例 id=1）：  
  `{"code":"SYSTEM_ERROR","success":false,"message":"系统繁忙，请稍后再试","data":null}`
- **影响**：依赖 **`GET /page-versions/{id}` 的版本详情/单独表单**无法在真实模式下工作；**页面版本编辑器**当前路由仅传 `versionId`，首屏加载走 **`GET /page-versions/{versionId}/node-tree`**，与该详情接口**解耦**（详情仍应尽快修复以备其它入口）。
- **建议后端修复方式**：排查详情接口 DB 映射、字段序列化、空指针等
- **前端当前处理方式**：`throwIfEnvelopeFailed` 后抛错；列表与其它可用接口仍可用；不 mock 掩盖

## 5. page-node / node-tree — 种子数据 component_key 大量为空（观测）

- **模块**：page-node（页面版本 / 模板节点树数据）
- **接口**：`GET /page-versions/{versionId}/node-tree`、`GET /page-templates/{id}/node-tree`（本机探测：`versionId=307`，`templateId=1`）
- **请求方式**：GET
- **请求体**：无
- **期望**：非 `fragment_ref` 的组件类节点应具备有效的 `component_key`（或与产品约定一致的可解析组件标识）
- **实际**：多条 `nodeType` 为 `container` / `component` 的节点 **`componentKey` 为 `null`**
- **影响**：前端 `validatePageNodeTree` 要求非片段引用节点必须存在且在 `component_def` 中的 `component_key`；用户直接「保存」可能被校验拦截；预览/属性 schema 与组件定义联动弱化
- **建议后端修复方式**：核对种子数据、模板生成链路及文档字段必填语义，补齐或明确「可为空」的契约
- **前端当前处理方式**：仅映射为 `component_key: null`；**`validatePageNodeTree` 等业务校验逻辑未改**

## 6. page-node — move 接口 sortOrder 可能未按请求持久化（观测）

- **模块**：page-node
- **接口**：`POST /page-nodes/{id}/move`
- **请求方式**：POST
- **请求体**（camelCase，实测）：`{"parentId":4070,"slotName":"main","sortOrder":10000}`
- **期望**：成功后 **GET `/page-versions/{versionId}/node-tree`** 中该节点 **`sortOrder`** 与请求一致（或与后端文档说明的重排规则一致）
- **实际**：HTTP 200 + `success:true`；响应体与随后 **GET node-tree** 中测试节点 **`sortOrder` 仍为 9999**（本机 `127.0.0.1:9876`，测试节点 id 4081，已于验证后删除）
- **影响**：依赖 move 精确排序的前端展示或与文档不符；联调难以确认 move 是否真正生效。**E-3 起编辑器在 move 后会 `loadEditor` 重拉 node-tree**，用户所见顺序以服务端为准；若服务端未持久化 sortOrder，将出现「操作未报错但顺序不变」的体验问题。
- **建议后端修复方式**：核对 move 持久化与返回值是否包含更新后的 `sortOrder`
- **前端当前处理方式**：调用 **`movePageNode`** 后 **重拉 node-tree**，返回值取重载后的节点；**`success:false` 时 throw**；不假本地移动成功。**E-4**：浏览器侧表现为移动后顺序可能与预期不符，以 Network 中 GET node-tree 为准。

## 7. page-node — copy 响应子树根节点 parentId 为空（观测）

- **模块**：page-node
- **接口**：`POST /page-nodes/{id}/copy`
- **请求方式**：POST
- **请求体**：`{}`（本机）
- **期望**：副本根节点 **`parentId`** 应与源节点一致（或与文档约定的插入位置一致），以便前端 **`buildNodeTree`** 正确挂树
- **实际**（本机 `127.0.0.1:9876`，源节点 **4083** 父为 **4070**）：响应 **`data.nodes[0].parentId` 为 `null`**，**`sortOrder`** 与源同为 **9999**
- **影响**：复制后 **`loadEditor`** 可能将副本视作根节点展示，与真实层级不一致；依赖 **`copyPageNode` + 重拉** 的树可能与业务预期不符
- **建议后端修复方式**：副本根节点应写入正确的 **`parentId` / `slot_name` / `sort_order`**
- **前端当前处理方式**：如实 **`mapPageNode`**；**`success:false` 时 throw**；不伪造父子关系

## 8. page-version — 整树 PUT 使用 snake_case 时父子关系丢失（本机观测）

- **模块**：page-version / page-node
- **接口**：`PUT /page-versions/{versionId}/node-tree`
- **请求方式**：PUT
- **请求体**：`{ "nodes": [...] }`，节点字段为 **snake_case**（含 **`parent_id`** 等），由 **`JSON.stringify`** 直接序列化前端 **`SavePageNodeTreePayload`**
- **期望**：与文档一致时能正确解析 **`parent_id`**，树结构保持不变
- **实际**（本机 **`127.0.0.1:9876`**，**versionId=307**）：HTTP 200 **`success:true`**，但响应 **`data.nodes`** 中 **全部节点 `parentId` 为 `null`**，树退化为多根/扁平，**结构被破坏**
- **影响**：旧前端若发送 snake_case，保存成功提示下实际毁坏版本树；编辑器 **`validatePageNodeTree`** 也可能异常
- **建议后端修复方式**：同时兼容 **snake_case** 与 **camelCase**，或与接口文档统一并明确绑定字段名
- **前端当前处理方式（E-5～E-6）**：真实 **`PUT`** **`/page-versions/{id}/node-tree`**、**`/page-templates/{id}/node-tree`**、**`/reusable-fragments/{id}/node-tree`** 均发送 **`buildSaveNodeTreeRequestBodyForBackend`**（camelCase 平铺 **`nodes`**）；store 仍构造 snake_case **`PageNodeTreeWriteItem`**，仅在 **`src/api/*`** 映射。**模板 / 片段 camelCase 整树 PUT** 已由 **E-6.2 curl** 验证闭环。

## 9. page-version — 整树 PUT 成功返回 id_map 并重映射节点 id（观测）

- **模块**：page-version / page-node
- **接口**：`PUT /page-versions/{versionId}/node-tree`
- **请求方式**：PUT
- **请求体**：camelCase 平铺 **`nodes`**（本机验证）
- **期望**：若文档约定保留客户端提交的 **`id`**，应稳定；否则应在文档说明全量替换语义
- **实际**：成功响应含 **`data.id_map`**（旧 id → 新 id），**`data.nodes`** 中 **id 全部更换**
- **影响**：外部引用 **`page_node.id`**、书签、审计日志若依赖稳定 id 会失效；编辑器 **`loadEditor`** 可收敛，但与其他系统联动风险高
- **建议后端修复方式**：明确产品语义（保留 id vs 每次重建）；若保留 id 则 **`id_map`** 仅用于临时 id 映射场景
- **前端当前处理方式**：**`mapPageNode`** 消费响应；**`replaceNodesFromResponse` / `loadEditor`** 以服务端返回为准；不伪造旧 id
