# 后端接口修复要求（YAAI 后台前端联调交付）

**文档用途**：供后端同学按优先级排查与修复；前端对照事实整理，**不修改《前端需求接口文档_v2.0.md》**。  
**联调日期**：2026-05-03  
**前端环境**：`VITE_USE_REAL_API=true`，`VITE_API_BASE_URL=/api`，浏览器请求形如 **`/api/...`**；后端监听 **`127.0.0.1:9876`**，**curl 直连业务路径不带 `/api`**。

---

## 1. 总体说明

### 1.1 前端真实联调环境

- 前端仓库：`yaai-lowcode-0423`，后台管理模块与低代码编辑器已接入 **HTTP 真实模式**。
- 开发态请求经 **Vite 代理** 转发至后端；**请勿用 `http://127.0.0.1:9876/api/...` 误判列表类接口**（该前缀直连常见 **`SYSTEM_ERROR`**，属路径误用）。

### 1.2 前端已完成的兼容策略（不要求后端为前端私自增字段）

- **`src/api/*`**：请求写出时对后端实际 **camelCase** 做适配；响应 **camelCase** → 映射为前端类型 **snake_case**（与数据库字段语义一致）。
- **`page-node` 整树保存**：三类 **`PUT …/node-tree`** 统一 **`buildSaveNodeTreeRequestBodyForBackend`**，**平铺 `nodes`**，**camelCase**；**JSON 字段保持字符串或 `null`，不做无谓 parse**。
- **真实失败**：**不 fallback mock**；**`success:false`** **throw**；UI **不假成功、不假删**。
- **页面版本编辑器**：单节点 **create/update/move/delete** 走 HTTP；**move/copy 后 `loadEditor`**；整树保存成功后再 **`loadEditor`**，缓解 **`id_map` 换 id**。
- **模板 / 片段编辑器**：结构变更以本地节点 + **整树保存**落库；保存成功后同样 **`loadEditor`**。

### 1.3 后端需要修复的范围

以下问题在 **正确路径**、**正确字段风格**下仍存在逻辑或契约风险，需后端修复或澄清。**详见下文 §2～§3 与 §5**。

### 1.4 本文不涉及

- 前端擅自新增的数据库字段或表。
- 修改接口契约文档正文（仅登记观测事实）。

## 2. 高优先级问题

### 2.1 node-tree：snake_case 请求 `success:true` 但毁树（数据安全问题）

| 项 | 说明 |
|----|------|
| **方法 / 路径** | **`PUT /page-versions/{versionId}/node-tree`** |
| **示例请求体** | **`{ "nodes": [ { "parent_id": 4070, "page_version_id": 307, ... } ] }`**（节点字段 **snake_case**） |
| **期望** | `success:true` 时 **不得丢失父子关系**；若不支持 snake_case，应 **`success:false`** 或明确拒绝 |
| **实际**（本机 **307**） | **`success:true`**，但响应 **`data.nodes`** 中 **`parentId` 全员 `null`**，树扁平化 |
| **对前端影响** | 误发旧格式会直接毁坏线上树 |
| **前端当前处理** | 真实写入一律 **camelCase** **`buildSaveNodeTreeRequestBodyForBackend`**；模板 / 片段路径同样 camelCase（**E-6.2 curl 已验证**） |

### 2.2 page-node move：`sortOrder` 未按请求持久化

| 项 | 说明 |
|----|------|
| **方法 / 路径** | **`POST /page-nodes/{id}/move`** |
| **示例请求体** | **`{"parentId":4070,"slotName":"main","sortOrder":10000}`** |
| **期望** | 成功后 **GET node-tree** 中该节点 **`sortOrder`** 与请求一致（或与文档说明的规则一致） |
| **实际** | **`success:true`**，但 **GET** 仍为旧 **`sortOrder`**（历史观测见 **`docs/后端待修复清单.md` §6**） |
| **对前端影响** | 用户感知「移动成功但顺序不变」 |
| **前端当前处理** | move 后 **重拉 node-tree**，以服务端为准 |

### 2.3 page-node copy：副本根节点 `parentId` 异常

| 项 | 说明 |
|----|------|
| **方法 / 路径** | **`POST /page-nodes/{id}/copy`** |
| **示例请求体** | **`{}`** |
| **期望** | 副本根 **`parentId`** 与源一致（或与文档约定一致） |
| **实际** | 响应中副本根 **`parentId` 为 `null`**（历史观测 **§7**） |
| **对前端影响** | 复制节点层级展示可能错误 |
| **前端当前处理** | copy 后 **`loadEditor`**，如实映射 |

### 2.4 component-def：`GET /component-defs/by-key/{componentKey}` SYSTEM_ERROR

| 项 | 说明 |
|----|------|
| **期望** | 返回组件定义（schema / default 等） |
| **实际** | **`success:false`** **`SYSTEM_ERROR`** |
| **对前端影响** | 无法按 key 单独拉取定义 |
| **前端当前处理** | 编辑器依赖 **列表缓存**；失败 **throw**，不 mock |

### 2.5 component-def：`DELETE /component-defs/{id}` SYSTEM_ERROR

| 项 | 说明 |
|----|------|
| **期望** | 删除成功或明确业务失败 |
| **实际** | **`SYSTEM_ERROR`**；历史上删除后 **GET 仍存在** |
| **对前端影响** | 删除闭环不可用 |
| **前端当前处理** | **不假删**，错误提示 |

### 2.6 page-version：`GET /page-versions/{id}` SYSTEM_ERROR

| 项 | 说明 |
|----|------|
| **期望** | 返回版本详情 |
| **实际** | **`success:false`** **`SYSTEM_ERROR`** |
| **对前端影响** | 依赖详情的表单 / 详情页不可用 |
| **前端当前处理** | 抛错 + 提示；编辑器首屏可走 **`GET …/node-tree`** |

## 3. 中优先级问题

### 3.1 menu：`POST /menus/{id}/move` SYSTEM_ERROR

- **影响**：菜单拖拽排序不可用。  
- **前端**：失败提示，不假成功（详见 **`docs/后端待修复清单.md` §1**）。

### 3.2 node-tree 保存 **`id_map` / 全量换 id** 的产品语义

- **现象**：成功 **`PUT …/node-tree`** 后返回 **`id_map`**，**节点 id 整体更换**（**§9**）。  
- **期望**：文档明确「保留客户端 id」还是「每次重建」；若保留 id，应避免无谓换 id。  
- **前端**：保存后 **`loadEditor`**，以外链稳定性仍需后端策略。

### 3.3 创建资源后 **`createdAt`/`updatedAt` 为 null**

- **观测**：部分 **POST** 模板 / 片段 / 页面（联调过程）返回 **`createdAt:null`**。  
- **期望**：若字段存在，应写入服务端时间或文档标明可为 null。  
- **影响**：列表展示与时间排序体验。

### 3.4 种子节点 **`componentKey` 大量为空**

- **观测**：**§5**，影响编辑器 **`validatePageNodeTree`** 与保存体验。  
- **期望**：数据或契约对齐。

## 4. 字段风格统一建议

- **文档 v2.0**：多以 **snake_case** 描述。  
- **运行态实测**：大量 **读写 body / 响应内嵌对象** 为 **camelCase**。  
- **前端**：已在 **`src/api/*`** 双向适配；**类型与表单层维持 snake_case 语义**（对齐数据库字段名）。  
- **建议后端**：
  - **要么** 更新接口文档，明确 camelCase 请求/响应规范；  
  - **要么** 接口统一 snake_case，且 **务必保证** snake_case **不会**再触发类似 **node-tree 毁树**（§2.1）的严重问题。
