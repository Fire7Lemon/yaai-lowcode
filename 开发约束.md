# 开发约束

## 一、当前项目基础

本项目不是从纯空目录开始，而是已经创建好了一个 Vue 项目。

后续开发必须在当前已有的 Vue 工程基础上继续进行，不允许：

- 擅自删除现有工程
- 擅自重建项目
- 擅自切换到其他前端框架
- 擅自替换 Vite / Vue / TypeScript 基础结构

如需调整目录结构，必须在不破坏当前工程可运行性的前提下进行。

---

## 二、技术栈约束

本项目技术栈固定为：

- Vue 3
- Vite
- TypeScript
- Element Plus

开发时必须遵守以下原则：

1. 使用 Vue 3 组合式 API
2. 使用 TypeScript 定义类型
3. 使用 Element Plus 完成后台界面基础组件
4. 使用模块化目录结构组织代码
5. 中文界面文案优先

未经明确要求，不要擅自引入新的大型前端框架或状态管理替代方案。

---

## 三、数据库一致性约束（最高优先级）

本项目的一切前端实现，必须严格以根目录中的《数据库设计_v6.0_平台型低代码版.md》为唯一数据模型标准。

这条是最高优先级要求。

任何时候都不允许：

- 擅自新增数据库表
- 擅自新增数据库字段
- 擅自修改字段名
- 擅自修改表关系
- 擅自修改核心模型含义
- 擅自扩展数据库文档中不存在的枚举值
- 擅自发明脱离数据库设计的第二套前端数据模型

所有实现都必须围绕数据库已有字段展开。

---

## 四、页面结构模型约束

本项目页面结构的唯一核心主线是：

**page -> page_version -> page_node**

必须明确：

- `page` 是页面元信息
- `page_version` 是页面版本
- `page_node` 是页面、模板、片段的统一节点树承载结构

不允许自行设计任何并行的第二套页面结构模型。

不允许使用与数据库设计无关的固定层级页面结构作为核心实现。

页面编辑器中的：

- 节点树
- 预览区
- 属性面板
- 节点新增 / 删除 / 排序
- 插槽
- 容器关系

都必须围绕 `page_node` 实现。

---

## 五、组件系统约束

组件能力必须围绕 `component_def` 组织。

必须严格基于以下字段理解组件系统：

- `component_key`
- `component_name`
- `component_group`
- `component_type`
- `icon`
- `is_container`
- `can_bind_data`
- `can_reuse_as_fragment`
- `prop_schema_json`
- `style_schema_json`
- `event_schema_json`
- `layout_schema_json`
- `allowed_child_types_json`
- `default_props_json`
- `default_style_json`
- `sort_order`
- `status`
- `remark`

不允许直接绕开 `component_def`，另行设计一套与数据库脱节的组件注册方式。

---

## 六、数据绑定约束

所有数据来源配置必须围绕 `data_binding` 组织。

必须严格基于以下字段理解数据绑定：

- `name`
- `binding_type`
- `source_key`
- `query_json`
- `field_map_json`
- `transform_json`
- `empty_state_json`
- `error_state_json`
- `cache_policy`
- `status`
- `remark`

不允许另行设计一套脱离 `data_binding` 的数据源系统。

---

## 七、类型定义约束

所有 TypeScript 类型必须来源于当前数据库设计。

要求：

1. 不允许在类型中出现数据库中不存在的核心字段
2. 不允许擅自修改已有字段含义
3. mock 数据必须与数据库字段保持一致
4. 页面表单类型、表格行类型、树节点类型、接口返回类型必须统一维护

建议建立独立的类型目录，例如：

- `src/types/page.ts`
- `src/types/page-version.ts`
- `src/types/page-node.ts`
- `src/types/component-def.ts`
- `src/types/data-binding.ts`
- `src/types/menu.ts`

---

## 八、接口组织约束

如果后端接口尚未完全提供，可以先建立 API 模块骨架与 mock 占位，但必须遵守以下原则：

1. API 模块命名必须围绕数据库实体命名
2. 不允许自己脑补与数据库不一致的接口对象结构
3. mock 数据字段必须严格来源于数据库
4. 如果接口细节不明确，先用 TODO 标记，不要擅自扩展字段

建议 API 按实体拆分，例如：

- `src/api/page.ts`
- `src/api/page-version.ts`
- `src/api/page-template.ts`
- `src/api/reusable-fragment.ts`
- `src/api/component-def.ts`
- `src/api/data-binding.ts`
- `src/api/page-node.ts`
- `src/api/menu.ts`

---

## 九、页面与目录结构约束

建议采用清晰的分层目录结构，例如：

- `src/layouts`
- `src/router`
- `src/views`
- `src/components`
- `src/api`
- `src/types`
- `src/mock`
- `src/utils`

页面目录建议围绕业务实体组织，而不是围绕随意的功能命名。

例如：

- `src/views/page`
- `src/views/page-version`
- `src/views/page-template`
- `src/views/reusable-fragment`
- `src/views/component-def`
- `src/views/data-binding`
- `src/views/menu`
- `src/views/editor`

---

## 十、页面编辑器约束

页面编辑器是本项目核心功能，必须遵守以下要求：

### 1. 结构固定
编辑器布局固定为：

- 左侧：节点树
- 中间：预览区
- 右侧：属性面板

### 2. 节点树核心来源
节点树只能来源于 `page_node`。

### 3. 预览区核心来源
预览区必须基于 `page_node + component_def + data_binding` 递归渲染。

### 4. 属性面板核心来源
属性面板必须编辑 `page_node` 的数据库字段，以及由 `component_def` 决定的 schema 相关配置。

### 5. 不允许另起模型
不允许为了让编辑器更容易实现，而私自构造一套只存在于前端、不对应数据库的核心结构。

允许做适度的前端视图层转换，但这种转换必须可逆，并且不能脱离数据库主结构。

---

## 十一、UI 与交互约束

本项目是后台系统，要求：

1. 风格简洁、清晰、专业
2. 表单、表格、树、抽屉、弹窗优先使用 Element Plus
3. 不要过度追求花哨视觉效果
4. 优先保证结构清晰、功能完整、字段准确
5. 所有界面命名、标签、列名尽量使用中文

---

## 十二、代码风格约束

要求：

1. 文件命名规范统一
2. 组件拆分合理
3. 单个页面不要堆积过多逻辑
4. API、类型、视图、组件分离
5. 能抽离的公共逻辑尽量抽离
6. 不要生成大量无意义样板代码
7. 不要生成与当前业务无关的复杂基础设施

---

## 十三、信息不足时的处理方式

如果开发过程中发现某个地方缺乏明确信息：

1. 优先查阅根目录所有文档
2. 采用最小合理实现
3. 用 TODO、placeholder、mock 占位
4. 不得自行新增数据库字段
5. 不得自行新增数据库表
6. 不得自行发明新的核心模型

---

## 十四、开发优先级

推荐优先级如下：

### 第一优先级
- 路由与布局
- 类型定义
- API 模块骨架
- 页面管理
- 页面版本管理

### 第二优先级
- 页面模板管理
- 可复用片段管理
- 菜单管理
- 组件定义管理
- 数据绑定管理

### 第三优先级
- 页面编辑器基础版
- 节点树
- 预览区
- 属性面板
- 节点操作

---

## 十五、最终目标

本项目最终目标不是做一个随意发挥的 Vue 后台 demo，而是：

**在现有 Vue 工程基础上，构建一个严格贴合 v6.0 数据库设计的平台型低代码后台前端。**
