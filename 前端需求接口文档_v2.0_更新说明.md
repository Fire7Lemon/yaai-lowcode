# 《前端需求接口文档》v1 → v2.0 更新说明

本文档说明 **`前端需求接口文档_v2.0.md`** 相对 **`前端需求接口文档_v1.md`** 的变更意图与条目摘要，便于评审与联调对照。**未改动 v1 原文**。

---

## 1. v2.0 的定位变化

| 项目 | v1 | v2.0 |
|------|-----|------|
| 字段依据 | 以前端当时写法为主 | **显式对齐**《数据库设计_v6.0_平台型低代码版.md》中 `menu`、`page`、`page_version`、`page_template`、`reusable_fragment`、`component_def`、`data_binding`、`page_node` |
| 范围说明 | 强调「已使用或明确需要」 | 保留该原则，并补充：**不覆盖 v6.0 内容类表**（新闻、轮播等）的后台接口 |
| 类型定义 | 多处仅写类型名、缺字段表 | **新增 §3 数据模型摘要**，按表列出字段，减少与库不一致的理解偏差 |

---

## 2. 全局约定（§2）更新

1. **响应信封**：v2 增加推荐的 `{ success, code, message, data }` 说明，与当前前端集成中对 `data` 的解析习惯一致。
2. **列表载荷**：在保留 `{ items, total, page, page_size }` 的同时，补充 **兼容形态**（如 `records`、`current`、`size`），避免与已对接后端冲突。
3. **`page_node` 归属**：单独写明与 v6.0 一致的 **三选一**（`page_version_id` / `template_id` / `fragment_id`）。

---

## 3. 接口总览新增或显式纳入的路径

| 变更类型 | 路径 | 说明 |
|----------|------|------|
| 新增 | `GET /pages/{id}/current-version` | 与 `page.current_version_id` 链路一致；草案中已有 |
| 新增 | `GET /page-versions/{id}` | 版本详情；草案中已有 |
| 新增 | `GET /component-defs/by-key/{componentKey}` | 按 key 取组件定义；草案中已有 |
| 新增 | `GET/POST/PUT/DELETE /menus/...`、`POST /menus/{id}/move` | 与 v6.0 `menu` 及菜单管理能力对齐；草案中已有 |
| 保留 | 其余 v1 编辑器与 CRUD 路径 | 未改名 |

---

## 4. 分模块字段与请求体修正（相对 v1）

### 4.1 页面（Page）

- **列表查询**：补充可选 **`title`**、**`current_version_id`**（与前端筛选及库字段一致）。
- **当前发布版本**：新增独立小节 **`GET /pages/{id}/current-version`**。

### 4.2 页面版本（PageVersion）

- **新建版本请求体**：由仅 `version_name`、`remark` 扩展为与库一致的 **`source_type`、`source_id`、`remark`**（`version_name` 可为 null）；并明确 **从模板创建走单独接口**。
- **版本列表查询**：补充 **`status`、`version_name`、`source_type`、`is_locked`**。
- **复制版本请求体**：补充可选 **`remark`**。
- **版本详情**：新增 **`GET /page-versions/{id}`**。
- **从模板创建**：正文推荐 **`application/json`** + **`template_id` / `version_name` / `remark`**（snake_case）；**兼容说明**中提及历史上可能出现的 **form-urlencoded + 驼峰**（与 v1「纯 JSON」表述不同，避免联调踩坑）。
- **发布返回**：写明 **`page` 至少包含 `id` 与 `current_version_id`**（可与完整 `Page` 兼容），避免强行要求完整 `Page` 与部分前端类型不一致。

### 4.3 页面模板（PageTemplate）

- **列表筛选**：增加 **`scene_type`**。
- **新建/更新体**：增加 **`scene_type`、`preview_image`、`remark`**，与 v6.0 表一致（v1 缺多项）。

### 4.4 可复用片段（ReusableFragment）

- **新建/更新体**：增加 **`remark`**；**`fragment_type`** 与库一致可为 null（文档中写 `string | null`）。

### 4.5 组件定义（ComponentDef）

- **列表筛选**：将 v1 中 **`component_key`** 为主的示例改为与前端 Query 及物料分组一致的 **`component_group`**（仍可按需支持按 key 精确筛，未单独列为必选）。
- **新建/更新体**：按 v6.0 **补全** `component_group`、`icon`、`can_reuse_as_fragment`、`sort_order`、`remark` 等；**删除** v1 中 **`description`**（**非** `component_def` 表字段）。
- **按 key 查询**：新增 **`GET /component-defs/by-key/{componentKey}`**。

### 4.6 数据绑定（DataBinding）

- **列表筛选**：将错误的 **`binding_name`、`binding_code`** 改为与库一致的 **`name`**，并补充 **`binding_type`、`source_key`**（v6.0 无 `binding_code`）。
- **新建/更新**：v1 仅写返回类型；v2 **补全请求体**，与 `DataBinding` 全字段一致。

### 4.7 菜单（Menu）

- v1 仅 **`GET /menus`**；v2 **补全**详情、新建、更新、删除、移动及 **Menu 全字段**请求体与列表筛选（含 **`url_type`**、分页可选）。

### 4.8 页面节点（PageNode）

- v1 多以类型名引用；v2 在 **§3.8** 给出 **与 v6.0 一致的字段表**，并明确 **CreatePageNodeInput / UpdatePageNodeInput / PageNodeTreeWriteItem** 边界。
- **可选用**：注明 **`POST /page-nodes/{id}/reference-fragment`** 为草案中的可选动作（非库表强制接口）。

## 5. 「需后端确认」清单（§14）

- 在 v1 的 10 条基础上，补充 **`from-template` 传输格式**、**删除菜单级联**、以及列表载荷 **`items` vs `records`** 等与 v2 约定直接相关的事项编号说明。

## 6. 未纳入 v2 的内容（有意为之）

- **未新增**任何 v6.0 未定义的库字段或新表接口。
